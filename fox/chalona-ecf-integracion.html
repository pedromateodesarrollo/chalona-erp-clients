Chalona ECF · Cliente Fox v1.0

Guía de integración del cliente Visual FoxPro chalona-ecf.prg con ERPs propios. Última revisión: abril 2026.

1. Resumen

chalona-ecf.prg es la librería VFP que integra un ERP Fox con el servicio Chalona ECF (e-CF DGII República Dominicana). Encapsula:

  • Login en el servidor (JWT Bearer).
  • Construcción del JSON DGII (encabezado, totales, detalle, retención, multimoneda).
  • Envío al endpoint envia_ecf.
  • Consulta de estado por e-NCF.
  • Sincronización masiva de comprobantes "en proceso" con sp_getapplock.
  • Descarga de documentos por rango de fechas (zip).
  • Manejo de errores con formulario opcional (ChalonaFormErrorEnvioEcf).

El acceso a datos del ERP está abstraído tras un driver inyectable. El driver por defecto (ChalonaEcfDriverSqlServer) lee de dbo.imtr, dbo.gastos, dbo.imtrd, etc. mediante la función global Request(). Otros ERPs implementan su propio driver con la misma interfaz (VFP es duck-typed; no se hereda nada).

TL;DR para integrarse
  1. Escribir una clase con los métodos del contrato del driver.
  2. loEcf = Createobject("ChalonaEcf").
  3. loEcf.SetDriver(Createobject("MiEcfDriver")).
  4. loEcf.Enviar(tcControl).

Entradas del motor:

  • ChalonaEcf.Enviar(tcControl) es el punto principal.
  • El motor llama a This.Driver.MetodoX() y nunca accede directamente a tablas dbo.* ni DBFs.
  • tcControl es opaco para el motor: el driver lo decodifica como quiera (id numérico, JSON, "tipo|numero", etc.).

3. Instalación y carga

Archivos físicos:

ArchivoRol
chalona-ecf.prg Motor + driver default (SQL Server) + helpers + clases UI.
chalona-ecf-loader.prg Loader: descarga la versión activa de data.fox_cliente_script y la compila como temporal para hot-reload.
chalona-ecf-driver-default.prg Plantilla standalone del driver default. Sólo referencia — el driver ya viene embebido en chalona-ecf.prg. Útil para clonar y adaptar.

3.1 Loader / hot-reload

El cliente productivo carga chalona-ecf-loader.prg, no el .prg directo. El loader:

  1. Consulta data.fox_cliente_script en el backend (filtrando por entorno según portal_dgii: testecf → entorno='test'; ecf → entorno='produccion').
  2. Escribe el script activo a un archivo temporal único.
  3. Hace SET PROCEDURE TO <tmp>.fxp ADDITIVE.
  4. El ERP llama Createobject("ChalonaEcf") y todo carga.

Publicar una versión nueva (desarrolladores Chalona):

# entorno=test (default)
./scripts/actualiza-cliente-fox.sh

# entorno=produccion
./scripts/actualiza-cliente-fox.sh --produccion
Nota de despliegue La tabla data.fox_cliente_script vive siempre en la BD producción (localhost:5432/produccion). El flag entorno es lo que decide qué ERPs reciben qué versión. Esto se ha pagado caro al publicar v11 a 5433 cuando debía ir a 5432; el script ya no acepta esa ambigüedad.

3.2 Configuración (objeto Cfg)

El motor es agnóstico al ERP host: opera contra un objeto de configuración con propiedades de tipo cadena. La configuración se inyecta de dos formas:

  1. ERPs con Public osis (Vicortiz): no se hace nada. Init llama ChalonaEcfConfigDesdeOsis() automáticamente y arma el objeto.
  2. ERPs sin osis (Alberto / otros): el ERP construye el objeto y lo inyecta vía chalonaSetConfig(loCfg) (loader) o loEcf.SetConfig(loCfg) (motor) antes de cualquier operación.

Propiedades esperadas en el objeto:

PropiedadUsoEjemplo
servidor_ecfURL base del servidor (override) https://ecf-service.vicortiz.com/
usuario_syncEmail de login del API user131086268@vicortiz.com
pass_syncClave del API—
portal_dgiiPortal DGII testecf | ecf
dgii_multimonedaSi "T", activa bloque OtraMoneda cuando imtr.moneda es divisa extranjera y imtr.tasa > 1 T / F

Si no se inyecta cfg ni hay osis, el loader cae a su URL hardcoded (https://ecf-service.vicortiz.com/) y entorno produccion; las credenciales quedan vacías y el login fallará.

4. Uso básico

Registrar el loader una sola vez al arrancar el ERP (o compilarlo dentro del .exe):

SET PROCEDURE TO chalona-ecf-loader.prg ADDITIVE

La configuración se especifica una sola vez por sesión. El loader la almacena en goChalonaEcfCfg (y el driver en goChalonaEcfDriver) y la re-inyecta automáticamente cada vez que se descarga / refresca el motor (versión desactualizada, hot-reload, etc.). No hay que repetir chalonaSetConfig en cada operación.

Dos pasos para operar:

* 1. Configurar.
LOCAL loCfg, loResp
loCfg = CREATEOBJECT("Empty")
ADDPROPERTY(loCfg, "servidor_ecf",     "https://ecf-service.vicortiz.com/")
ADDPROPERTY(loCfg, "usuario_sync",     "user...@vicortiz.com")
ADDPROPERTY(loCfg, "pass_sync",        "***")
ADDPROPERTY(loCfg, "portal_dgii",      "ecf")
ADDPROPERTY(loCfg, "dgii_multimoneda", "F")

* (Opcional) driver custom — sin esto: default SqlServer.
chalonaSetDriver(CREATEOBJECT("AlbertoEcfDriver"))

chalonaSetConfig(loCfg)

* 2. Operar.
loResp = chalonaEnviaEcf("")
loResp = chalonaSincronizaEstados()
loResp = chalonaDescargaDocumentosEcf("2026-01-01", "2026-04-30")

Ventajas del mecanismo

  • Cliente mínimo: solo el loader (~300 líneas) vive en el ERP. Toda la lógica DGII se descarga del servidor.
  • Hot-reload por versión: cuando el servidor publica un script nuevo, el siguiente chalonaEnviaEcf detecta fox_cliente.version_desactualizada, redescarga y reintenta — sin reinstalar nada en el cliente.
  • Multi-ERP, misma API: cualquier host (Vicortiz, Alberto, otros) usa las mismas funciones top-level. El driver inyectado adapta el origen de datos (SQL Server, DBFs, otro esquema).
  • Sin INI ni configuración en disco: cfg llega por objeto desde el ERP. URL del servidor cae al hardcoded del loader si no se inyecta nada.
  • Aislamiento: el ERP no instancia clases internas ni conoce la firma del motor; sólo cuatro funciones públicas.

5.1 Properties

Init PROCEDURE

Si This.Cfg es .Null. al instanciar, lo arma con ChalonaEcfConfigDesdeOsis(). Aplica servidor_ecf, usuario_sync, pass_sync, portal_dgii a las properties de la clase. Crea ChalonaEcfDriverSqlServer si This.Driver está nulo.

SetConfig(toCfg) PROCEDURE

Inyecta el objeto de configuración. Llamar inmediatamente después de Createobject("ChalonaEcf"). Re-aplica BaseUrl, Usuario, Clave y Portal al setear. Para ERPs sin Public osis es la única manera de configurar el motor.

LOCAL loCfg
loCfg = CREATEOBJECT("Empty")
ADDPROPERTY(loCfg, "servidor_ecf",     "https://ecf-service.vicortiz.com/")
ADDPROPERTY(loCfg, "usuario_sync",     "...")
ADDPROPERTY(loCfg, "pass_sync",        "...")
ADDPROPERTY(loCfg, "portal_dgii",      "ecf")
ADDPROPERTY(loCfg, "dgii_multimoneda", "F")
loEcf.SetConfig(loCfg)

SetDriver(toDriver) PROCEDURE

Reemplaza el driver. Llamar inmediatamente después de Createobject("ChalonaEcf").

loEcf.SetDriver(CREATEOBJECT("AlbertoEcfDriver"))

Enviar(tcControl) → ChalonaResponse PROCEDURE

Flujo completo: login → build JSON → POST envia_ecf → persistir respuesta vía driver (GuardarRespuestaEnvio / MarcarErrorEnvio).

Retorno: objeto ChalonaResponse con .ok (L), .message (C), .data (O), .rawBody (C), .requestBody (C).

Datos relevantes en .data al éxito:

  • numero — eNCF asignado.
  • estado — "Aceptado" / "Rechazado" / "En Proceso".
  • codigo_seguridad, fecha_firma, timbre.
  • secuencia_utilizada, momento (ISO).

ConsultarEstado(tcComprobantesJson) → ChalonaResponse

tcComprobantesJson = JSON array de eNCFs: '["E32...","E31..."]'. Llama consulta_estado en el backend.

SincronizarEstadosEnProceso() → ChalonaResponse

Toma todos los comprobantes con respuesta_estado = 'En Proceso' (via driver), consulta DGII en lotes de 100 e-NCF y persiste resultado. Usa mutex (SyncIntentarLock) para evitar concurrencia.

DescargarDocumentos(tcDesde, tcHasta, tcTipos) → ChalonaResponse

Pide GETFILE('zip'), llama ecf_documentos_list y guarda los XMLs/JSONs en un ZIP. Si el usuario cancela el diálogo, retorna ok=.T. con message="cancelado".

6. Contrato del driver

Cualquier objeto VFP con los siguientes métodos sirve como driver. No hay clase abstracta: VFP es duck-typed. Si un método falta, el motor fallará en runtime al invocarlo.

Patrón cursor + finalize Este es el único caso donde el motor escribe en un cursor del driver. El driver puede implementarlo de dos formas:
  • Cursor temporal sintético (default SqlServer): el driver crea un Create Cursor local independiente. En SyncFinalizar recorre y emite UPDATEs.
  • Cursor sobre tabla real: el driver hace USE de la tabla con buffering, y en SyncFinalizar hace TABLEUPDATE.

6.2 Métodos del contrato

Lecturas

MétodoRetornaNotas
CargarMaestro(tcControl) "curChalMae" o "" Cabecera del documento. Cursor vacío = "no existe" → motor termina sin error.
EsGastos(tcControl) → L .T./.F. True si el control es de compras (41/43).
CargarDetalle(tcControl) "curChalDet" o "" Solo ventas. En gastos el motor sintetiza una línea y NO llama esto.
CargarFiscalVence(tcTipoEcf) "curChalFis" o "" Fallback de FechaVencimientoSecuencia. Cursor vacío = motor abortará si fechavencencf también está vacío en el maestro.
CargarEmpresa() "curChalEmp" o "" Datos del emisor.
CargarSuplidorRncNombre(tcCodigo) "curChalSup" o "" Solo gastos. Fallback cuando el maestro no trae RNC.
CargarTerceroExtranjero(tcCodigo, tlEsGastos) "curChalCli" o "" Cliente o suplidor para flag extranjero. Si tlEsGastos=.F. devolver también rnc/nombre.
CargarReferenciaImtr(tcOcontrol) "curChalRef" o "" Lookup del NCF referenciado para NC/ND.

Origen del control

ContarOrigen(tcControl) → Empty Objeto con propiedades .imtr y .gastos (counts). Motor lo usa para _DocOrigen (decide imtr vs gastos vs ambiguous).

Persistencia de respuesta

GuardarRespuestaEnvio(tcControl, loData, tlEsGastos) → L Persistir numero (eNCF), estado, momento, codigo_seguridad, fecha_firma, timbre, secuencia_utilizada, estado_descripcion en la tabla origen.
MarcarErrorEnvio(tcControl, tcMensaje, tlEsGastos) → L Persistir solo respuesta_mensajes (acotado: 254 imtr / 250 gastos).

Sync masivo

SyncIntentarLock() → N Retorna ≥0 si tomó el lock; <0 si ya está tomado; -99 si error de infraestructura.
SyncLiberarLock()Libera el lock.
SyncListarPendientes() → C Cursor READWRITE con shape ampliado (ver 6.1 curChalonaEncfEnProceso). Trae los "En Proceso" con e-NCF asignado. Las columnas de respuesta vienen vacías; el motor las llena.
SyncListarDuplicados() → C Cursor (control) con duplicados cross-table. Vacío si no aplica.
SyncFinalizar() → L El motor lo llama tras escribir los resultados de DGII en el cursor. El driver decide qué hacer: UPDATE batch a la tabla origen, flush via TABLEUPDATE, append a tabla de log, no-op, etc.

6.3 Notas VFP importantes

Parámetros omitidos llegan como .F. En VFP, los LParameters que no se pasan llegan como .F. (false), no como .Null.. Alltrim(.F.) revienta. Defendé al inicio: 
If Vartype(tcControl) # "C"
   tcControl = ""
Endif
No usar RETURN dentro de TRY/CATCH VFP no lo permite (error 2060 "no manejado"). Usar un flag y un If después del EndTry.

7. Driver default: ChalonaEcfDriverSqlServer

Implementación que viene con el motor. Asume:

  • Conexión SQL Server activa, accesible vía Request(tcSql, tcAlias) (función global definida por el ERP).
  • Tablas: dbo.imtr, dbo.imtrd, dbo.mercs, dbo.gastos, dbo.fiscal, dbo.empresa, dbo.clientes, dbo.suplidor.
  • Mutex via sp_getapplock sesión-scoped.

El archivo chalona-ecf-driver-default.prg contiene una copia standalone de la clase para que sirva de plantilla / referencia. Está embebida también dentro de chalona-ecf.prg.

Manejo de version_desactualizada (obligatorio)

El servidor rechaza los envíos cuando la programación del cliente Fox no es la versión activa (o no reporta versión, típico de motores viejos en caché). El flujo esperado es:

  1. Automático: el loader detecta el código, redescarga la versión activa, la instancia en memoria y reintenta el mismo envío. Tu ERP no ve nada — loResp.ok vuelve .T..
  2. Si el retry no resuelve (loader viejo sin retry, descarga fallida, o el servidor sigue rechazando): loResp.ok=.F. con loResp.message = "fox_cliente.version_desactualizada". En ese caso el motor v47+ ya devuelve un texto claro en respuesta_mensajes / messagebox: "Cliente ECF desactualizado: CIERRE y vuelva a ABRIR el sistema…".

Tu ERP debe detectar este código y guiar al usuario a reiniciar (cerrar y reabrir la aplicación fuerza la descarga del motor nuevo desde el arranque). Patrón recomendado:

loResp = chalonaEnviaEcf(lcControl)
IF !loResp.ok
   IF ATC("fox_cliente.version_desactualizada", loResp.message + loResp.rawBody) > 0
      * Version vieja y el auto-update no resolvio. Pedir reinicio.
      MESSAGEBOX( ;
        "El modulo de facturacion electronica se actualizo." + CHR(13) + ;
        "Por favor CIERRE y vuelva a ABRIR el sistema para continuar.", ;
        48, "Actualizacion requerida")
      * No reintentar en loop: el motor nuevo solo carga al reabrir.
   ELSE
      * Otros errores: mostrar loResp.message normalmente.
      MESSAGEBOX(loResp.message, 16, "Error al enviar e-CF")
   ENDIF
ENDIF

Nota: reintentar el mismo chalonaEnviaEcf en bucle NO ayuda dentro de la misma corrida: el motor cacheado (goChalonaEcf) solo se reemplaza al reabrir el sistema o cuando el loader logra la redescarga. Si tu loader es reciente (con retry), el caso 2 casi no ocurre; el snippet cubre los clientes con loader viejo.

Helpers de logging:

  • ChalonaEcfLogError(tcMsg, tcControl, tcExtra) — error textual.
  • ChalonaEcfLogException(tcCtx, tcControl, toEx, tcExtra) — desde Catch To.
  • ChalonaEcfSaveUltimoJson(tcControl, lcDocJson) — guarda JSON enviado.

11. Troubleshooting

"Class not found: ChalonaEcfDriverSqlServer"

El loader cargó solo parte del prg, o la versión activa de data.fox_cliente_script es vieja. Republicar (./scripts/actualiza-cliente-fox.sh) y reiniciar el ERP.

"Argument list too long" al publicar

El script actualiza-cliente-fox.sh debe pasar el SQL via stdin, no -c. Versión actualizada (≥abril 2026) ya lo hace.

DGII rechaza con código 613 ("no pueden reemplazarse entre ellos")

Tipo 31 referenciando otro e-CF que no es FC electrónica. El motor ya omite la referencia en este caso, pero verificar que el driver no esté forzando ocontrol con un eNCF inválido.

DGII rechaza con código 11204 (TipoMoneda inválido)

El XSD no acepta TipoMoneda="RD". Si moneda del maestro es DOP/RD/peso, el motor NO emite OtraMoneda; si llega "RD" igualmente, revisar imtr.moneda (debe ser vacío, "DOP" o divisa real).

"No se puede sincronizar respuesta; control ambiguo"

ContarOrigen retornó imtr ≥ 1 AND gastos ≥ 1. Indica que el mismo tcControl existe en ambas tablas. Bug de datos: corregir antes de seguir.

El motor llama Request() pero el ERP no la tiene

El driver default (SqlServer) usa Request(tcSql, tcAlias). Si tu ERP no la define, escribir un driver custom y NO usar el default.
